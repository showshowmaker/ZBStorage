可以，这样改更干净。

既然你现在明确要求：

`一个文件的对象要么全部归档，要么全部不归档`

那就不需要 `archived_object_count` 这种“部分完成计数”了。  
这相当于把归档语义从“对象集合状态”收紧成“文件原子状态”。

所以 `ArchiveFileMeta` 可以改成更简单的版本：

```cpp
struct ArchiveFileMeta {
    uint64_t inode_id{0};
    std::string disk_id;
    uint64_t file_size{0};
    uint32_t object_count{0};
    uint64_t last_access_ts_ms{0};
    FileArchiveState archive_state{FILE_ARCHIVE_PENDING};
    uint64_t version{0};
};
```

这样这个 store 的职责就变成：

- 记录文件最近访问时间
- 记录文件对象数
- 记录文件级状态
- 落 WAL / snapshot，和对象级 store 一样

不再负责记录“已经归档了几个对象”，因为你们不允许出现“部分归档是有效状态”。

下面是按这个约束整理后的“文件级归档状态持久化”代码实施清单。

## 6. 文件级候选筛选逻辑改造

当前你已经在 `CollectFileArchiveCandidates(...)` 里按对象元数据临时聚合。  
补完 `ArchiveFileMetaStore` 后，改成直接读文件级 store。

### 筛选条件
只保留：

- `archive_state == pending`
- `last_access_ts_ms > 0`
- `now_ms - last_access_ts_ms >= min_age_ms`

### 排序规则
只按最近访问时间：

- `last_access_ts_ms` 越小，优先级越高
- 相同则按 `inode_id`

### 不再做的事
- 不再扫描对象级 `archive_state`
- 不再统计已归档对象数
- 不再允许“部分归档也算某种稳定状态”

## 7. 归档过程中的文件级状态流转

这里是这次改造的核心。

### 在 MDS 归档一整个文件前
当 `OpticalArchiveManager` 取到文件候选并准备开始 stage 整文件对象时：

- 调目标 data node 的 `UpdateFileArchiveState(inode_id, disk_id, ARCHIVING, version)`

修改点在：

- [OpticalArchiveManager.cpp](c:/Users/w1j2h/Desktop/AllZB/ZBPro/ZBStorage/src/mds/archive/OpticalArchiveManager.cpp)

建议在确定整文件任务要进入 batch 之前调用。

### 整文件全部归档成功后
当该文件全部对象都写入 optical 成功，并且对象级状态回写成功后：

- 调 `UpdateFileArchiveState(..., ARCHIVED, version)`

### 整文件任一对象失败
如果文件级任务中任一对象失败：

- 回退成 `PENDING`

这正符合你要求的语义：

`文件要么全部归档，要么全部没归档`

所以：
- 不需要 `partial_failed`
- 不需要 `archived_object_count`

## 12. 一句话原则

这次的设计原则可以定成：

`文件级候选由文件级最近访问时间决定，文件级归档状态由整文件任务驱动，文件状态只允许 pending / archiving / archived 三态，不允许部分归档成为稳定状态。`
