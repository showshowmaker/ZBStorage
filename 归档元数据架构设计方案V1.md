# 归档元数据架构设计方案 V1

## 1. 背景与目标

当前主项目中的 `mds` 模块负责在线元数据权威存储，底层使用 RocksDB，主要管理以下数据：

- `inode` 元数据
- `dentry` 元数据
- 文件位置元数据
- 归档状态与归档任务状态

现有主路径适合在线元数据，但不适合直接承载“1000 亿级文件元数据”的冷数据场景。当前独立的 `masstree` 目录已经验证了一种基本思路：

- 内存中维护一个有序索引
- 磁盘上顺序落页保存真实记录
- 查询时先查索引，再查页内记录

这套思路适合继续演进为“归档元数据子系统”，服务于大规模冷命名空间。

V1 目标如下：

1. 支持 1000 亿级冷文件元数据的长期存储。
2. 支持按 namespace 批量导入，单次导入量可达 1 亿文件级别。
3. 支持 `Lookup`、`Getattr`、`Readdir` 三类核心查询。
4. 保持现有在线 MDS 主路径不被大改，采用“路由到热存储/冷存储”的方式接入。
5. 第一版不要求支持频繁在线修改，优先支持“批量生成 + 批量导入 + 只读查询”。


## 2. 非目标

V1 明确不做以下能力：

- 不支持归档命名空间上的高频在线 `Create/Mkdir/Rename/Unlink/Rmdir`
- 不支持跨 namespace 的强事务
- 不支持单条记录级别的细粒度 WAL 回放
- 不支持多副本一致性复制
- 不支持把全部元数据常驻内存

如果后续需要小规模增量更新，另行增加 delta 层，不在 V1 范围内。


## 3. 设计原则

### 3.1 热冷分层

- 热元数据继续由现有 RocksDB MDS 承担。
- 冷元数据由新的 `ArchiveMetaStore` 承担。
- MDS 作为统一入口，根据 namespace 或 path 路由到不同底层。

### 3.2 不可变代次

- 每次导入产生一个新的 `generation`。
- `generation` 一旦发布后只读。
- 发布通过 manifest 原子切换完成。

这样可以大幅降低恢复、并发和一致性复杂度。

### 3.3 稀疏索引 + 顺序段文件

- 真实元数据不直接全部放入 Masstree。
- Masstree 只做页级稀疏索引。
- 真正的记录存储在排序好的段文件中。

### 3.4 双表模型

归档元数据必须至少拆成两类：

- `inode table`
- `dentry table`

不能只做一张 path 表，否则会破坏现有 `inode` 直查和目录枚举能力。


## 4. 总体架构

```text
                   +----------------------+
Client RPC ------> |      MdsService      |
                   | Lookup/Getattr/...   |
                   +----------+-----------+
                              |
                    namespace / path route
                              |
              +---------------+----------------+
              |                                |
              v                                v
   +---------------------+         +------------------------+
   |   HotMetaStore      |         |   ArchiveMetaStore     |
   | RocksDB, online     |         | cold namespace only    |
   +---------------------+         +-----------+------------+
                                               |
                                  +------------+-------------+
                                  |                          |
                                  v                          v
                        +------------------+       +------------------+
                        | Inode SegmentSet |       | Dentry SegmentSet|
                        +--------+---------+       +--------+---------+
                                 |                          |
                                 v                          v
                        +------------------+       +------------------+
                        | Masstree Index   |       | Masstree Index   |
                        | sparse page idx  |       | sparse page idx  |
                        +--------+---------+       +--------+---------+
                                 \__________________________/
                                              |
                                              v
                                   +----------------------+
                                   | Segment/Page Reader  |
                                   | page cache + verify  |
                                   +----------------------+
```


## 5. 模块划分

### 5.1 MetaStoreRouter

位置建议：

- `src/mds/storage/MetaStoreRouter.h`
- `src/mds/storage/MetaStoreRouter.cpp`

职责：

- 对 MDS 屏蔽热存储和冷存储的差异
- 根据 namespace、path 或 inode 所属范围选择底层
- 暴露统一接口供 `MdsServiceImpl` 调用

建议接口：

```cpp
class MetaStoreRouter {
public:
    bool GetInode(uint64_t inode_id, zb::rpc::InodeAttr* attr, std::string* error);
    bool GetDentry(uint64_t parent_inode, const std::string& name, uint64_t* inode_id, std::string* error);
    bool Readdir(uint64_t parent_inode,
                 const std::string& start_after,
                 uint32_t limit,
                 std::vector<zb::rpc::Dentry>* entries,
                 std::string* next_token,
                 bool* has_more,
                 std::string* error);
    bool ResolvePath(const std::string& path, uint64_t* inode_id, zb::rpc::InodeAttr* attr, std::string* error);
};
```

### 5.2 ArchiveNamespaceCatalog

位置建议：

- `src/mds/archive_meta/ArchiveNamespaceCatalog.h`
- `src/mds/archive_meta/ArchiveNamespaceCatalog.cpp`

职责：

- 记录哪些 namespace 已归档
- 记录 namespace 当前指向的 generation
- 保存 namespace 对应的 shard 信息
- 承担“发布新 generation”的原子切换入口

建议底层：

- 第一版直接存到现有 RocksDB
- catalog 数据量很小，不需要额外复杂化

### 5.3 ArchiveMetaStore

位置建议：

- `src/mds/archive_meta/ArchiveMetaStore.h`
- `src/mds/archive_meta/ArchiveMetaStore.cpp`

职责：

- 对归档命名空间提供只读查询
- 管理 inode/dentry 两类 SegmentSet
- 打开 manifest
- 装载 Masstree 稀疏索引
- 调用页缓存与段文件读取器

建议接口：

```cpp
class ArchiveMetaStore {
public:
    bool OpenNamespace(const std::string& ns, std::string* error);
    bool GetInode(uint64_t inode_id, zb::rpc::InodeAttr* attr, std::string* error);
    bool GetDentry(uint64_t parent_inode, const std::string& name, uint64_t* inode_id, zb::rpc::InodeType* type, std::string* error);
    bool Readdir(uint64_t parent_inode,
                 const std::string& start_after,
                 uint32_t limit,
                 std::vector<zb::rpc::Dentry>* entries,
                 std::string* next_token,
                 bool* has_more,
                 std::string* error);
};
```

### 5.4 SegmentSet

职责：

- 管理某个 generation 下的一组段文件
- 每种表各自独立：
  - `inode SegmentSet`
  - `dentry SegmentSet`
- 根据 shard 和 segment_id 定位页

### 5.5 PageReader / PageCache

职责：

- 从段文件中读取页
- 校验页头、校验和
- 维护热点页缓存
- 向上提供页内二分和顺序遍历能力

### 5.6 ArchiveMetaBuilder

位置建议：

- `tools/archive_meta/archive_meta_builder.cpp`

职责：

- 接收元数据生成结果
- 做排序、分页、段构建、索引构建、manifest 输出
- 在全部文件构建成功后发布 generation


## 6. 数据模型

### 6.1 逻辑表

V1 使用两张逻辑表：

#### inode table

- key: `inode_id`
- value: `InodeAttr` 的序列化结果

#### dentry table

- key: `(parent_inode, name)`
- value: `child_inode_id + type`

### 6.2 key 编码

不建议直接沿用在线 RocksDB 的字符串 key：

- `I/<inode_id>`
- `D/<parent_inode>/<name>`

归档层建议改成二进制有序 key，降低体积和比较成本。

#### inode key

```text
[table_id=1][inode_id:u64_be]
```

#### dentry key

```text
[table_id=2][parent_inode:u64_be][name_len:u16_be][name_bytes]
```

说明：

- `table_id` 用于隔离 inode 和 dentry
- `u64_be` 保证按字典序与数值序一致
- `name_len` 可以让页内解析更直接

### 6.3 value 编码

#### inode value

- 直接存 `zb::rpc::InodeAttr::SerializeToString()`

优点：

- 可直接复用现有 RPC 结构
- 编码逻辑简单
- 与现有 MDS 数据兼容性最好

#### dentry value

建议存自定义紧凑结构：

```text
[child_inode_id:u64_le][type:u8]
```

原因：

- `Readdir` 只需要 `name + inode_id + type`
- 避免为 dentry 再套 protobuf


## 7. 物理存储格式

### 7.1 generation 目录结构

建议根目录：

```text
<archive_root>/
  catalog/
  namespaces/
    <namespace_id>/
      current
      generations/
        <generation_id>/
          manifest.json
          inode/
            shard_000/
              segment_000000.dat
              segment_000001.dat
              index.dat
            shard_001/
          dentry/
            shard_000/
              segment_000000.dat
              segment_000001.dat
              index.dat
```

说明：

- `current` 指向当前 generation id
- 每个 generation 完整自描述
- 每类表按 shard 拆分

### 7.2 segment 文件

每个 segment 是若干页顺序拼接而成。

建议页大小：

- 第一版使用 `64KB`

原因：

- 大于 4KB 时页级索引项数量明显减少
- 页内二分成本仍可接受
- 顺序读放大适中

### 7.3 page 格式

```text
PageHeader
SlotOffsets[]
RecordData...
PageTrailer
```

建议字段：

#### PageHeader

- magic
- version
- table_id
- page_id
- record_count
- data_region_size
- min_key_len
- max_key_len
- reserved

#### RecordData

每条记录：

```text
[key_len:u16][value_len:u32][key][value]
```

#### PageTrailer

- crc32

### 7.4 稀疏索引文件

Masstree 索引项只针对每个页写一条或少量条目：

```text
key   = page_max_key
value = (segment_id, page_id)
```

说明：

- 查询时先用 `search/scan`
- 找到不小于目标 key 的页
- 再读页做二分

这与当前 `masstree` 原型的总体思路一致，但要从实验代码改成正式格式。


## 8. 查询设计

### 8.1 Getattr(inode_id)

流程：

```text
MdsService
  -> MetaStoreRouter::GetInode
  -> ArchiveMetaStore::GetInode
  -> inode Masstree sparse index search
  -> locate page
  -> read page from cache or disk
  -> page binary search
  -> parse InodeAttr
  -> return
```

复杂度：

- Masstree 查找：`O(log N_pages)`
- 页内二分：`O(log N_records_per_page)`

### 8.2 Lookup(path)

沿用当前主项目的语义：逐级解析 path。

流程：

```text
current = root
for each component in path:
  dentry = GetDentry(current, component)
  current = dentry.child_inode
final inode = GetInode(current)
```

说明：

- 第一版不做 path 全路径索引
- 先保持行为与现有 `ResolvePath` 一致
- 后续如有必要，可增加 path cache

### 8.3 Readdir(parent_inode)

底层必须支持 prefix scan，这也是 Masstree 相比 hash 索引的核心价值之一。

#### 建议修改 RPC

当前 `Readdir` 一次返回全部子项，不适合大目录。

V1 建议增加分页：

```proto
message ReaddirRequest {
  string path = 1;
  uint32 limit = 2;
  string start_after = 3;
}

message ReaddirReply {
  MdsStatus status = 1;
  repeated Dentry entries = 2;
  bool has_more = 3;
  string next_token = 4;
}
```

#### 底层流程

```text
prefix = [table=2][parent_inode]
start_key = prefix + encoded(start_after)
Masstree.scan(start_key)
  -> 顺序读取后续页
  -> 页内顺扫
  -> 直到 key 不再属于当前 parent_inode 或达到 limit
```

### 8.4 文件位置查询

V1 设计范围聚焦于 namespace 元数据本身。  
如果冷 namespace 仍需保留 `DiskFileLocation/OpticalFileLocation`，建议两种方式：

#### 方案 A

- 仍保留在现有 RocksDB
- key 为 `inode_id`
- 只在 `GetFileLocation` 时额外查

#### 方案 B

- 在归档层增加 `location table`
- 与 inode 同 generation 一起导入

V1 推荐先用方案 A，降低改造面。


## 9. 批量导入设计

### 9.1 输入

导入器输入的是“已经生成好的元数据”，而不是在线 MDS 的实时操作流。

建议输入拆分为两路：

- inode records
- dentry records

### 9.2 构建流程

```text
metadata generator
  -> emit inode stream
  -> emit dentry stream
  -> external sort by key
  -> build pages
  -> build segments
  -> build Masstree sparse indexes
  -> write manifest.tmp
  -> fsync
  -> rename to manifest
  -> catalog publish generation
```

### 9.3 为什么不建议在线逐条插入

如果把 1 亿文件作为 live mutable 树逐条写入，会遇到以下问题：

- 内存占用不可控
- split 开销高
- 崩溃恢复复杂
- 与热查询相互干扰

因此 V1 明确采用：

- 离线排序
- 一次构建
- 原子发布

### 9.4 namespace 导入单位

建议把“单个 namespace”作为最小导入和发布单位。

优点：

- 边界清晰
- 失败回滚简单
- 权限和可见性管理简单


## 10. 分片设计

### 10.1 必须分片

1000 亿级别不能设计成单实例、单索引、单目录。

V1 至少采用两级分片：

- namespace 级分片
- table 内 shard 分片

### 10.2 inode 分片

建议按 `inode_id` range 分片：

```text
inode shard = inode_id / shard_range
```

优点：

- 实现简单
- 数据分布可预测

### 10.3 dentry 分片

建议按 `parent_inode` range 分片，而不是按 name hash。

原因：

- `Readdir(parent_inode)` 必须尽量落单 shard
- 按 parent 范围分片更适合目录枚举

### 10.4 shard 数量

第一版建议可配置，例如：

- 每个 namespace 默认 64 个 dentry shard
- 每个 namespace 默认 32 个 inode shard

具体数值由构建器根据数据规模决定。


## 11. 索引装载与内存策略

### 11.1 装载原则

- 只装 Masstree 稀疏索引
- 不装全量记录
- 页数据按需读取

### 11.2 PageCache

建议使用 LRU 或 CLOCK：

- key: `(namespace, generation, table, shard, segment_id, page_id)`
- value: page bytes

### 11.3 内存预算

需要支持配置：

- index memory budget
- page cache budget

V1 不允许无上限增长。


## 12. 一致性与恢复

### 12.1 发布语义

一次 generation 发布必须满足：

1. 所有 segment 文件落盘成功
2. 所有 index 文件落盘成功
3. manifest 写完并 fsync
4. catalog 切换 current generation

在此之前，新 generation 不可见。

### 12.2 manifest 内容

建议包含：

- namespace id
- generation id
- create timestamp
- inode shard 列表
- dentry shard 列表
- 每个 shard 的 segment 列表
- 页大小
- 记录数
- 校验和
- 构建版本

### 12.3 恢复流程

服务启动时：

```text
load catalog
  -> find current generation
  -> load manifest
  -> open all shard indexes
  -> lazy open segments
```

如果索引文件损坏，可做降级恢复：

- 扫描 segment 页头
- 重新构建稀疏索引

V1 可先不自动实现重建，只保留工具支持。


## 13. 与现有 MDS 的集成

### 13.1 接入原则

不直接在 `MdsServiceImpl` 中散落判断逻辑。  
统一在 `MetaStoreRouter` 或等价抽象层中完成路由。

### 13.2 需要接管的主流程

#### Lookup

- 保持 RPC 不变
- 内部改用 router 的 `ResolvePath`

#### Getattr

- 保持 RPC 不变
- 内部改用 router 的 `GetInode`

#### Readdir

- 建议改成分页 RPC
- 内部改用 router 的 `Readdir`

### 13.3 热冷路由策略

V1 建议按 namespace 路由，不按单个 inode 动态判定。

示例：

- `/hot/*` -> RocksDB
- `/archive/ns_0001/*` -> ArchiveMetaStore

路由元数据由 `ArchiveNamespaceCatalog` 管理。


## 14. 现有 masstree 原型的改造建议

当前 `masstree` 目录可以继续利用，但不能直接作为生产模块使用。

需要改造的点：

### 14.1 保留的部分

- Masstree 底层并发有序索引实现
- `search/scan` 能力
- 基于页最大 key 的稀疏索引思路

### 14.2 需要重写的部分

- 实验性质的日志页格式
- ad-hoc 的 key/value 拼接方式
- 测试程序中的线程池与文件写入逻辑
- 恢复逻辑中的目录遍历方式

### 14.3 建议抽象

新增正式封装：

```cpp
class SparsePageIndex {
public:
    bool Open(const std::string& index_path, std::string* error);
    bool SearchPage(const KeyView& key, PagePointer* out, std::string* error);
    bool ScanPages(const KeyView& start, uint32_t limit, std::vector<PagePointer>* out, std::string* error);
};
```


## 15. RPC 与接口变更建议

### 15.1 Readdir 分页化

这是 V1 最重要的接口变更之一。

建议新增：

- `limit`
- `start_after`
- `has_more`
- `next_token`

归档路径上的 `next_token` 不应再暴露为“最后一个名字”本身，而应作为不透明游标返回。
V1 当前实现为 `generation_id + last_name` 的封装 token，翻页时先校验代次一致，再在 generation 内部继续按 `last_name` seek。
这样 generation 发布切换发生在两次分页请求之间时，第二页会明确返回 token 失效，而不会静默跨到新 generation 继续遍历。

### 15.2 可选新增按 inode 查询接口

如果当前客户端已有 `Getattr(inode_id)`，则无需新增。  
否则建议对归档层尽量使用 inode 直查，减少路径逐级展开开销。

## 17. V1 实施阶段

### 阶段 1：只读单机版本

目标：

- 单 namespace
- 单 generation
- 单机多 shard
- 支持 `Getattr` / `Lookup` / 分页 `Readdir`

不做：

- 发布切换
- catalog 持久化
- 增量导入

### 阶段 2：可发布版本

目标：

- 增加 manifest
- 增加 namespace catalog
- 支持 generation 原子切换
- 接入 MDS 路由

### 阶段 3：工程化版本

目标：

- page cache
- 配置化内存预算
- 校验工具
- 导入校验
- 监控指标


## 18. 风险与应对

### 风险 1：Readdir 语义不改会拖垮系统

问题：

- 大目录无法一次性完整返回

应对：

- V1 必须引入分页 `Readdir`

### 风险 2：把 Masstree 当全量主存索引会内存爆炸

问题：

- 1000 亿级别不可能全部常驻内存

应对：

- 只做页级稀疏索引

### 风险 3：单 namespace 过大导致单 shard 热点

问题：

- 大目录可能导致 dentry shard 局部热点

应对：

- 按 `parent_inode` range 分 shard
- 后续预留二级分桶能力

### 风险 4：导入过程不可恢复

问题：

- 构建中断可能留下脏文件

应对：

- 使用临时 generation 目录
- 仅在 manifest 完整写入后发布

### 风险 5：与现有 MDS 逻辑耦合过深

问题：

- 后续维护复杂

应对：

- 路由层抽象统一封装
- `MdsServiceImpl` 只面向统一接口


## 19. V1 决策总结

V1 的关键决策如下：

1. 不用当前 `masstree` 原型直接承载主项目元数据，而是保留其“稀疏有序索引”思路。
2. 冷元数据采用“immutable generation + 稀疏 Masstree 索引 + 顺序段文件”。
3. 数据至少拆成 `inode table` 和 `dentry table` 两张逻辑表。
4. 查询只实现 `Lookup`、`Getattr`、分页 `Readdir`。
5. 导入采用离线排序构建，不做在线逐条写。
6. 现有 RocksDB MDS 不替换，只增加归档元数据路由层。


## 20. 建议的下一步

建议按以下顺序推进：

1. 确认 V1 是否接受“归档 namespace 只读”约束。
2. 确认 `Readdir` 是否允许做分页协议升级。
3. 先落一版 `archive_meta` 目录结构和核心类空壳。
4. 先实现只读单机版：
   - inode lookup
   - dentry lookup
   - 分页 readdir
5. 再实现 builder、manifest 和 generation publish。

## 22. 当前实现状态评估

截至当前代码状态，归档链路已经不是“纯方案阶段”，而是已经具备一条可运行的最小闭环：

- 已有 `ArchiveImportService`
- 已有 `ArchiveGenerationPublisher`
- 已有 `ArchiveNamespaceCatalog`
- 已有 `ArchiveMetaStore`
- 已有 `MetaStoreRouter`
- 已有 `ImportArchiveNamespace` RPC

但它距离“可长期上线稳定运行”仍然存在 4 类缺口：

1. 发布语义还不是成品级，缺少持久化级别的发布保证、回滚和垃圾回收语义。
2. generation cache 已有基本淘汰，但还缺外部可观测性和更精确的资源统计。
3. 导入链路已经服务化，但还不是后台 job 化的批量导入系统。
4. inode 路由已经有 bucket shortlist + bloom filter，但还不是强定位。

## 24. 实施顺序与当前代码改造范围

建议实施顺序如下：

1. 先补发布语义里的持久化和失败补偿能力。
2. 再把导入服务 job 化。
3. 再补 generation cache 的 metrics 和排障接口。
4. 最后优化 inode 强定位。

当前代码改造先聚焦第 1 项，当前已完成第一阶段：

- 已为 segment、index、bloom、manifest 增加显式落盘同步
- 已为目录 rename 后增加父目录同步
- 已在 route 发布失败时增加补偿清理
- 已引入 `current generation pointer`，并接入 catalog 读路径

当前仍未完成：

- 孤儿 generation 的识别与自动清理策略

这样做的目的，是优先消除“坏 generation 可见”与“发布失败留下脏状态”这类正确性风险。
