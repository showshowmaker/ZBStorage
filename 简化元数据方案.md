可以，按你的设想改的话，建议走下面这套方案。

目标

MDS 不再维护完整 file->object 布局，只保留轻量入口。
每个真实节点本地维护“该节点承载文件”的完整元数据（自描述）。
客户端先从 MDS 拿“文件锚点（anchor）”，后续读写走锚点节点。
一、架构改造（核心）

引入 FileAnchor 概念。
MDS 只存：inode_id -> anchor_node_id/anchor_addr/disk_id/file_meta_version。
引入 NodeFileCatalog（真实节点本地元数据目录）。
节点存：文件元数据、对象映射、版本、校验、冷热状态。
FUSE 读写改为两跳。
先 MDS 取 anchor，再直接请求 anchor 节点拿布局并读写。
二、元数据边界调整

MDS 保留：
命名空间：目录、dentry、inode 基本属性。
文件入口：FileAnchor（轻量）。
PG 视图与节点健康（用于 placement 决策/校验）。
真实节点新增权威元数据：
FileMeta {inode_id, size, object_unit_size, version, mtime...}
ObjectMap {object_index -> object_id + replicas + checksum + state}
ReplicaIntent/CommitLog（用于崩溃恢复）
约束：
单文件单元数据主（anchor）写入。
其他副本节点只做数据副本和必要索引，不做主元数据决策。
三、协议改造

MDS 新增/替换接口（mds.proto）：
GetFileAnchor(inode_id) -> FileAnchor
UpdateFileAnchor(inode_id, expected_ver, new_anchor)（迁移/故障切主）
real node 新增接口（real_node.proto）：
GetFileMeta(inode_id)
AllocateFileWrite(inode_id, offset, size)（返回 object 计划）
CommitFileWrite(inode_id, txid, expected_ver)
ResolveFileRead(inode_id, offset, size)（返回可读对象与副本）
保留兼容窗口：
旧 AllocateWrite/ResolveLayout/CommitLayoutRoot 暂时可保留为 shim，内部转发到新链路。
四、读写流程（改后）

读流程：
FUSE -> MDS GetFileAnchor
FUSE -> anchor ResolveFileRead
FUSE -> data node ReadObject（按返回副本读）
写流程：
FUSE -> MDS GetFileAnchor
FUSE -> anchor AllocateFileWrite（拿 object 分配计划）
FUSE -> data nodes WriteObject
FUSE -> anchor CommitFileWrite（版本+原子提交）
anchor 成功后异步/同步回写 MDS inode size/version（轻量）
五、一致性与恢复

anchor 节点本地元数据必须 WAL + snapshot（你现有思路可复用）。
CommitFileWrite 使用 txid + expected_version 幂等提交。
anchor 故障时：
MDS 标记 anchor 不健康；
从副本中选新 anchor；
新 anchor 通过本地自描述元数据重建 NodeFileCatalog 后接管。
MDS 与 anchor 之间允许短暂最终一致，但以 anchor 版本为真值。

七、实施顺序（建议）

Phase A：先做单副本文件（replica=1）闭环。
MDS 仅存 anchor，文件读写全走 anchor 节点元数据。
Phase B：扩展多副本。
anchor 管理副本写入与提交，增加重试/repair。

八、验收标准

MDS 无完整 file->object 布局写入（只写 anchor + inode 基础）。
文件可仅靠 anchor 节点本地元数据完成读写。
anchor 故障可切换并恢复读写。
单副本与多副本路径都通过 e2e。
